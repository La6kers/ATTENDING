Critical — Fix Now (3)

These are actual code bugs or correctness issues regardless of deployment stage.

Resolved Bug
#1 — LabOrdersController Returns Guid.Empty as Provider ID on Claim Parse Failure
backend/src/ATTENDING.Orders.Api/Controllers/LabOrdersController.cs

GetCurrentUserId() returns Guid.Empty if the JWT sub claim is missing or malformed. Lab orders are then created with an invalid provider ID. This is a real bug — even in dev mode with the DevAuthHandler, a misconfigured claim will silently produce bad data in the database.

✓ Fixed in commit cdbce26 — Now throws InvalidOperationException with descriptive message including both sub and oid claim values.
Resolved Bug
#2 — Drug Interaction Service Can Produce Duplicate Entries
backend/src/ATTENDING.Domain/Services/DrugInteractionService.cs

Both the direct drug-drug check and class-based interaction check can add the same interaction pair. Deduplication uses exact string match, but the same pair can appear as ("Warfarin","Aspirin") and ("Aspirin","Warfarin") or with different casing. When displayed to a provider, the same interaction appears twice, creating noise in clinical decision-making.

✓ Fixed in commit cdbce26 — Added NormalizeDrugPair() helper and HashSet<(string, string)> deduplication with alphabetically-sorted, lowercased pair keys.
Resolved Bug
#3 — Assessment Status Transition Not Validated (Frontend API)
apps/provider-portal/pages/api/assessments/[id].ts — PATCH handler

The PATCH endpoint accepts any status string from the client and writes it to the database after .toUpperCase(). There's no validation that the value is a valid status or that the transition is legal. This means a client can set status: "BANANA" and it'll be stored, breaking any code that expects enum values. This is a real bug regardless of auth context.

✓ Fixed in commit cdbce26 — Added VALID_STATUSES enum check (400 on invalid) and ALLOWED_TRANSITIONS state machine (409 on illegal transition: CREATED→IN_PROGRESS→COMPLETED→REVIEWED).

#4 — send-to-provider Broadcasts to ALL Active Providers
apps/patient-portal/pages/api/send-to-provider.ts

When a patient sends a message, it creates notifications for ALL active providers (up to 50). Even in dev/demo mode, this produces wrong behavior — the feature should demonstrate correct routing. When a clinic connects, this will immediately leak messages to wrong providers.

✓ Fixed — Now resolves target provider via: (1) explicit assignedProviderId in request, (2) patient's latest assessment assignment, (3) demo fallback to first provider with console warning.

apps/provider-portal/pages/api/notifications/index.ts

The GET handler queries prisma.user.findFirst({ where: { role: 'PROVIDER' } }) instead of using the actual user making the request. When multiple providers exist in demo data, all providers see the same notification inbox. This makes multi-provider testing impossible.

✓ Fixed — Added resolveProviderId() helper using getServerSession(). Falls back to first provider only in dev mode when NEXTAUTH_ENFORCE is not set. Returns 401 in production without session.
Resolved Correctness
#6 — TieredClinicalIntelligence Marks AI Tier as Executed When Stubbed
backend/src/ATTENDING.Infrastructure/Services/TieredClinicalIntelligenceService.cs

Tier 2 (Cloud AI) is TODO-stubbed but tiersExecuted.Add("Tier2_CloudAi") runs anyway. Response metadata says AI was consulted when it wasn't. Providers or demos that inspect the metadata will show false information about AI involvement.

✓ Already resolved in codebase — Tier2 is now only added to tiersExecuted after successful execution. Tier0 (PureDomain) runs unconditionally as expected.

#7 — Dashboard Shows Empty State on API Failure (No Error UI)
apps/provider-portal/pages/dashboard.tsx

Promise.allSettled is used correctly, but failed requests fall back to empty arrays with no error indication. During demos, if the .NET backend isn't running, the dashboard silently shows zero patients instead of a helpful error message.

✓ Fixed — Added dataErrors state tracking per API source. Shows red error banner with specific messages (e.g. "Could not load assessments (Network error). Is the backend running?") instead of empty queue.
Resolved Correctness
#8 — Care Gap Rule Engine O(n*m) Performance
backend/src/ATTENDING.Domain/Services/CareGapRuleEngine.cs

LastScreening() uses LINQ Where + Contains on every measure evaluation. With 10+ quality measures and growing patient history, this becomes quadratic. Not a problem with demo data (small N), but will hit hard when a clinic connects with real patient panels of 2,000+.

✓ Fixed — Added lazy-initialized ScreeningIndex dictionary (built once on first access). Exact matches are O(1); substring matches fall back to single linear scan. Eliminates repeated LINQ chains.
Resolved Correctness
#9 — DiagnosticLearning N+1 Query in GetAllLatestSnapshots
backend/src/ATTENDING.Infrastructure/Repositories/DiagnosticLearningRepository.cs

GetAllLatestSnapshotsAsync runs one query to get distinct types, then N additional queries per type. Should be a single window-function query. Will be slow when recommendation types grow.

✓ Fixed — Replaced N+1 loop with single LINQ query: GroupBy subquery kept as IQueryable (not materialized), then joined back with .Where(s => latestDates.Any(...)) for a single round-trip to the database.
Resolved Correctness
#10 — InputSanitization May Skip Record Init-Only Properties
backend/src/ATTENDING.Application/Behaviors/InputSanitizationBehavior.cs

Sanitizer uses reflection to find backing fields for C# record properties. If the backing field naming convention doesn't match (varies by compiler version), sanitization silently skips the property. No warning logged.

✓ Fixed — Added LogWarning when backing field not found, including property name, type name, and a note that content was not sanitized. Summary warning logged at end with total skipped count.

Deferred — Address at Integration Time

These items are intentional scaffolding for the current plug-and-play stage. They'll need attention when connecting to real clinics/EHRs but are not bugs today.

Auth Bypass in Dev Mode

Middleware allows unauthenticated access when NODE_ENV=development. This is intentional for local development. When deploying to staging with clinic data: ensure NEXTAUTH_ENFORCE=true is set. Consider adding a startup check that throws if neither Azure AD nor enforce flag is configured in non-localhost environments.

Docker-compose Credentials

Hardcoded SQL Server password in docker-compose.yml. Fine for local dev (file is gitignored for .env.local, and this is the local dev compose). Before staging: switch to ${DB_SA_PASSWORD} env var substitution.

Mock Transcription Fallback

Returns mock transcript when no Whisper/Azure Speech key configured. Correct behavior for demos. Before real patients: fail explicitly if transcription service not configured and the feature is enabled.

Drug Interaction Null Stub

NullExternalDrugInteractionClient returns empty results when external API not configured. Local rules still run. Before clinical use: log warning that only local rules are active. Consider requiring external API in production config.

Token Revocation Refresh Window (8 hours)

Token is checked against DB only on refresh, not every request. Acceptable for dev/demo. Before multi-provider production: reduce to 15-minute tokens with refresh rotation, or check on every request with caching.

CSRF Validation Not Enforced

CSRF infrastructure exists (token generation, cookie setting) but not validated on mutation endpoints. Not exploitable without real authenticated sessions. Before clinic integration: add verifyCsrfToken() to all POST/PATCH/DELETE handlers.

Audit Logging Incomplete

Only PATCH assessment handler creates audit logs. Read operations not logged. Not needed for demo data. Before PHI access: add audit logging to all API handlers per HIPAA § 164.312(b).

42 CFR Part 2 Enforcement

IsPartTwoProtected flag exists on behavioral health records but isn't enforced at query level. Correct for demo. Before behavioral health clinic integration: add repository-level filter for Part 2 records.

Emergency Access Tenant Filter Bypass

Emergency facesheet assembler uses normal tenant-filtered queries. Emergency access by design should bypass tenant filters. Before emergency access feature goes live: use IgnoreQueryFilters() for emergency repositories, validated by access code.

Synthetic/Demo Data Separation

Demo patients distinguished only by "DEMO-" MRN prefix. Fine for current stage. Before mixing real and demo data: add IsDemo boolean flag to Patient entity with global query filter.

Strengths (17 Patterns)

Domain Model Quality: A

Rich aggregate roots with proper encapsulation. Factory methods enforce invariants (Patient.Create(), LabOrder.Create()). Value objects with clinical significance (VitalSigns calculates MAP, Shock Index, SIRS criteria automatically). Strong enum modeling for clinical concepts. Domain events on all significant state changes.

Clean Architecture: A-

Proper dependency inversion — domain has zero infrastructure references. Application layer uses CQRS with MediatR pipeline behaviors (validation, logging, sanitization). Infrastructure implements all domain interfaces. API layer is thin with proper middleware composition.

Multi-Tenant Isolation: A

EF Core global query filters on OrganizationId. _adminContextEnabled pattern for background jobs (not null bypass). EnableAdminContext() requires explicit opt-in with documented reason. Tenant ID auto-set from JWT claims.

Event-Driven Architecture: A

30+ domain event types covering full clinical lifecycle. IHasDomainEvents interface with ChangeTracker collection. Events dispatched in SaveChangesAsync for transactional consistency. Emergency/red flag events wire to SignalR for real-time notifications.

Clinical Safety Patterns: A-

Drug interaction checking (local rules + external API interface). Red flag evaluator with severity levels. Care gap rule engine with quality measures. Behavioral health screening with C-SSRS suicide risk detection. 42 CFR Part 2 flagging for substance use records. Emergency access with audit logging.

CQRS Pipeline: A

MediatR pipeline behaviors: FluentValidation → InputSanitization → Logging → Handler. Commands and queries properly separated. Result<T> pattern for explicit error handling without exceptions in business logic.

Soft Delete & Concurrency: A

All entities soft-delete by default with IsDeleted, DeletedAt, DeletedBy. Optimistic concurrency via RowVersion on every entity. Audit timestamps (CreatedAt, ModifiedAt, CreatedBy, ModifiedBy) auto-set.

BFF Pattern: A-

Next.js API routes proxy to .NET backend with Prisma fallback. Circuit breaker pattern in backend proxy. Auth tokens bridged from NextAuth to .NET + SignalR. Clean separation between portal UI and business logic.

XState Assessment Machine: A

18-phase OLDCARTS assessment modeled as type-safe state machine. Red flag detection at each phase. Context accumulates clinical data through transitions. Clean separation between UI interaction and clinical logic.

Distributed Lock Service: A

Redis implementation with SET NX EX pattern and Lua atomic release. In-memory fallback using SemaphoreSlim(1, 1) for development. Clean interface (IDistributedLock) with IsAcquired check pattern.

EF Core Resilience: A

EnableRetryOnFailure(maxRetryCount: 3, maxRetryDelay: 30s). Transaction management via IUnitOfWork. CommitTransactionAsync properly separates save from commit. Database fixture uses unique GUIDs per test instance.

DevOps & CI/CD: A

Blue-green deployment with Azure App Service slots. k6 smoke testing against staging before swap. Automated rollback on health check failure. Multi-stage Docker builds. OpenTelemetry distributed tracing with Jaeger/OTLP export. GitHub Actions with matrix builds for parallel CI.

Plug-and-Play Organization Model: A

Organization entity with onboarding state machine. EHR connector configuration per organization. Data mode switching (Demo → Hybrid → Live). Connector health checking. Designed for multi-tenant SaaS from day one.

Diagnostic Learning Engine: A-

Outcomes tracking, learning signals, accuracy snapshots. Calibration adjustment per guideline. Confidence decay over time. Self-improving clinical AI foundation. Well-modeled even though AI tier is currently stubbed.

Security Infrastructure: B+

CSRF token pair generation (HMAC-based). AES-256-GCM encryption utilities. PHI-safe logging policy. Rate limiting infrastructure. CSP with nonce support. All the pieces are built — they just need to be wired together at integration time.

Ambient Scribe Architecture: A-

Full recording lifecycle (consent → capture → transcribe → generate note → review). Consent tracking before recording. Audio blob cleanup scheduling. Multi-service transcription (Whisper + Azure Speech). Generated SOAP notes with approval workflow.

Behavioral Health: A

Validated screening instruments (PHQ-9, GAD-7, C-SSRS, AUDIT, DAST-10). Automatic scoring with clinical thresholds. Suicide risk detection triggers emergency pipeline. 42 CFR Part 2 protected record flagging. Provider review workflow with action documentation.
